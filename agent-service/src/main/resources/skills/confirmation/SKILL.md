# Skill: confirmation — 敏感操作前的二次确认

## 何时触发

你准备调用任何**修改/删除/发送/支付/分享**类 tool 之前。包括但不限于:

- 删除文件、删除照片、删除消息
- 发送消息、发邮件、发微信
- 修改设置、改密码、改权限
- 任何涉及钱(支付、转账)
- 把数据发给第三方(分享、上传到外部服务)

**只读类 tool**(列照片、读通知、看屏幕)不需要触发本 skill。

> 当前 tool 集还没有这类破坏性 tool,但本 skill 会在它们被加进来时立刻生效 — 提前埋好行为约束。

## 执行流程

1. **不要直接调 tool**。先把要做的事用一句话总结,**包含关键参数**。
   - 反例:"我要发一条消息,确认?"
   - 正例:"我要发微信给『张三』,内容『今晚 8 点见』,确认?"
2. **等用户的肯定回复**(嗯/好/确认/yes/对/go),再调 tool。
3. 如果用户答得含糊("可能吧"、"看你"),**当作否定** — 再问一次更具体的版本,或者请用户直接指明。
4. 用户取消(不/算了/no),立刻停止,不要找借口接着调。

## 与 server 端机制的关系

某些 tool 在 device 上声明了 `confirm_required: true`,server / device-hub 会自带一层二次确认 UI。**那一层是基础设施级保护,本 skill 是 LLM 行为级别的兜底** — 即使某个 tool 漏配了 `confirm_required`,你的行为仍然要做这层确认。

两层都做,不要因为"反正 server 会拦"就跳过 LLM 层确认 — 用户体验差(被 server 弹窗中断而不是 agent 自己说清楚要做什么)。

## 不要过度触发

- 用户**已经在同一句话里明确指令**("把那条消息删了"),且参数无歧义 — 简短复述一次"删除消息 ID xxx,执行"然后就调,不用反复来回确认。
- 用户**连续多次同类操作**(批量删除时已经确认过第一条),后续同类的可以直接执行,但批次结束要总结做了什么。

确认是为了防止误触发,不是为了仪式感。
